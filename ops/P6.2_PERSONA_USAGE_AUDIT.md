# P6.2 用户画像 Prompt 链路审计

## 1. aiPersona 生成链路

### 完整路径

```
用户提交五道题
    ↓
GET /api/diagnosis/session/[id]/result
    ↓
calculateResult(answers) → result.persona（规则结果）
    ↓
if (isAIEnabled()) {
  // 构建诊断答案文本
  diagnosisText = Q1:${答案}...Q5:${答案}
  
  // 构建 Prompt（lib/ai.ts chat()）
  personaPrompt = [
    {
      role: "system",
      content: "你是一个专业的商业咨询师。根据用户的诊断答案，生成一句精准的用户画像描述。
规则：
1. 画像必须基于用户已作答的答案，未作答的题目请忽略，不要提及"未作答"
2. 一句话说明用户的核心痛点和特征
3. 格式：「用户类型 + 痛点描述 + 期望」
..."
    },
    {
      role: "user", 
      content: "用户诊断答案如下：\n${diagnosisText}\n\n请根据以上答案，生成一句精准的用户画像描述。"
    }
  ]
  
  personaResponse = await chat(personaPrompt)
  // chat() 调用 lib/ai.ts 中的 DeepSeek API，timeoutMs=20000, max_tokens=300
    ↓
  aiPersona = stripped(personaResponse.content)
}
    ↓
finalPersona = aiPersona ?? result.persona  // 优先 LLM，否则用规则
    ↓
NextResponse.json({
  result: { ...result, persona: finalPersona },  // 合并进 result.persona
  aiPersona: aiPersona ?? null,                   // 独立字段也返回
  fields,
  workflow
})
    ↓
前端页面
    ↓
POST /api/execute/generate
  body.userPersona = diagnosisResult.persona  // ← 来自 finalPersona
```

### 关键代码片段

**Result API（route.ts:48-72）**：
- 调用方式：`const personaResponse = await chat(personaPrompt)`
- action 参数：**无 action 参数**，是独立的 LLM 调用
- system prompt：商业咨询师角色 + 5条规则 + 3个示例
- max_tokens：300（Round 18 修复后）

## 2. aiPersona 当前是否被实际使用

**结论：已使用（部分）**

### 使用点① — Result 页面展示
`app/result/page.tsx:178`：
```tsx
<p className="text-sm md:text-lg font-medium leading-relaxed">{persona}</p>
```
- 这里渲染的是 `data.result.persona`，即 **finalPersona**（`aiPersona ?? result.persona`）
- 如果 aiPersona 非空 → 用户看到 LLM 生成的用户画像 ✅
- 如果 aiPersona 为 null → 用户看到规则生成的 persona ✅

### 使用点② — Execute API
`app/execute/page.tsx:131-138`：
```tsx
const body: Record<string, string | undefined> = {
  action: actionId,
  type: diagnosisResult.type,
  sessionId,
  userPainPoint: diagnosisResult.painPoint,
  userPersona: diagnosisResult.persona,  // ← finalPersona
  workflowKey,
};
```
Execute API（`/api/execute/generate:317`）：
```ts
if (userPersona) ctxParts.push(`目标用户：${userPersona}`);
const diagnosisContext = ctxParts.length > 0 ? `${ctxParts.join("，")}。` : "";
const enrichedPrompt = `${diagnosisContext}${prompt ?? ""}`.trim();
```
- **aiPersona（finalPersona）通过 `userPersona` 传入了 execute API**
- 作为诊断上下文的一部分注入到图片生成 prompt 中 ✅

### 未使用点 — aiPersona 独立字段
Result API 额外返回了独立字段 `aiPersona: aiPersona ?? null`，但：
- `result/page.tsx` **没有**引用 `data.aiPersona`
- `execute/page.tsx` **没有**引用 `data.aiPersona`

目前独立字段存在但未被前端直接使用，只通过 `result.persona` 间接使用。

## 3. 影响范围分析

| 环节 | aiPersona 状态 | 结果 |
|------|---------------|------|
| Result 页面展示 | aiPersona ?? result.persona | ✅ 降级正常 |
| Execute API 上下文 | diagnosisResult.persona（finalPersona） | ✅ 降级正常 |
| aiPersona 独立字段 | 已返回但未使用 | ⚠️ 冗余字段 |

**核心影响**：
- 如果 AI 生成失败（超时/报错）→ `aiPersona = null` → 降级到规则 `result.persona` → 流程继续，无阻塞
- Prompt 质量问题（"未作答"误判）→ 影响用户看到的画像描述质量，以及传入 execute API 的上下文质量

## 4. 建议：投入产出比

### 是否值得优化？
**值得，但优先级中低**

#### 理由：
1. **已有降级链路**：AI 生成失败不会阻塞业务，规则结果兜底
2. **已修复 P6.1 超时问题**：timeout 20s + max_tokens 300，大幅降低超时概率
3. **Prompt 质量是残余问题**：Round 10 修复了"未作答"误判，但 SYSTEM_PROGRESS.md 记载"质量待优化"，说明仍有误判情况

#### 优化收益：
- 提升 result 页面展示的用户画像可读性（用户更愿意留下联系方式）
- 提升 execute API 的 prompt 上下文质量 → 图片生成效果更贴合用户需求

#### 建议方案：
1. 先收集一批真实 aiPersona 输出样例（E2E 测试）
2. 分析残留的"未作答"等低质量 case，对 Prompt 做二次优化
3. 独立 `aiPersona` 字段可考虑移除，避免前端困惑（已合并到 `result.persona`）

**预计投入**：1-2 小时
**预期收益**：中等（主要提升用户体验和 lead 转化置信度）
