# P6.2 用户画像 Prompt 优化建议

## 1. 当前 Prompt 原文

```typescript
const personaPrompt = [
  {
    role: "system" as const,
    content: `你是一个专业的商业咨询师。根据用户的诊断答案，生成一句精准的用户画像描述。

规则：
1. 画像必须基于用户已作答的答案，未作答的题目请忽略，不要提及"未作答"
2. 一句话说明用户的核心痛点和特征
3. 格式：「用户类型 + 痛点描述 + 期望」
4. 语言简洁专业，适合直接展示给用户
5. 不要解释，直接输出画像句子
6. 不要输出任何推理过程，不要输出任何标签

示例：
"内容引流效率低、有产品但缺流量的电商商家，期望找到高效的内容生产方式"
"客服响应慢、重复问题多的客服瓶颈商家，期望自动化处理提升效率"
"数据整理繁琐、渴望自动化提效的效率型商家，期望用工具替代重复劳动"`,
  },
  {
    role: "user" as const,
    content: `用户诊断答案如下：\n${diagnosisText}\n\n请根据以上答案，生成一句精准的用户画像描述。`,
  },
];
```

**五道题诊断题目（lib/diagnosis.ts）：**

| 题号 | 问题 |
|------|------|
| Q1 | 你现在最头疼的事情是什么？（内容/流量/效率/不明确） |
| Q2 | 你每个月花多少时间在这件事上？（紧急度） |
| Q3 | 你的主要销售平台是？（平台渠道） |
| Q4 | 你最希望通过AI解决什么问题？（核心诉求） |
| Q5 | 你现在的图片内容是怎么来的？（内容生产现状） |

---

## 2. 三条具体问题

### 问题一：Prompt 缺少业务背景上下文，画像难以直接用于 CTA 个性化

**现状：**
Prompt 只说"你是专业的商业咨询师"，没有明确告知 LLM 用户是**服装电商商家**（天猫/抖音/小红书），也没有告知诊断类型（流量型/客户型/效率型/不明确型）与画像之间的关系。

这导致 LLM 生成的画像是泛化的"电商商家"描述，无法直接用于：
- **CTA 文案个性化**（如流量型用户 vs 效率型用户需要不同话术）
- **工作流推荐**（画像结论没有与 action、workflowKey 挂钩）
- **跟进话术**（没有体现平台特性：天猫重产品图、抖音重内容、小红书重种草）

**影响：**
生成的用户画像句子看起来通顺，但前端使用时发现"画像跟 CTA 没关系"，无法驱动转化。

---

### 问题二：示例数量不足，且未覆盖"不明确"诊断类型

**现状：**
Prompt 只给了 3 个示例，全部是流量/客户/效率三个方向，**没有任何示例覆盖诊断类型为"不明确"（`type: unclear`）的用户**。

结合五道题的评分逻辑（traffic/customer/work 三维），如果用户每个维度得分都低（≤2分），会被判定为 `unclear`。这类用户在真实 E2E 中占比不低，但他们最需要的是**明确的引导**，而非泛泛的"你想引流获客"描述。

**影响：**
`unclear` 类型用户得到的画像会是模糊、无方向的空话，无法帮助前端做分流和 CTA。

---

### 问题三：Prompt 没有约束输出与诊断类型的语义关联，画像结构不稳定

**现状：**
规则 3 要求"用户类型 + 痛点描述 + 期望"三段式，但：
- "用户类型"是什么？LLM 可能输出"流量型商家"、"内容创作商家"、"效率型商家"——没有任何约束保证与 `result.type`（`traffic`/`customer`/`work`/`unclear`）一致
- Q2（时间投入/紧急度）和 Q4（AI 期望）的信息没有被要求整合进画像
- 150 tokens 的 `max_tokens` 限制了输出长度，但 prompt 没有明确告知 LLM 控制在多长范围内

这导致不同题目组合可能生成结构迥异的画像，有的包含平台信息有的没有，无法形成产品侧需要的统一画像格式。

---

## 3. 优化方向（具体怎么改）

### 优化一：补充业务背景上下文，明确用户身份和画像用途

**改动方案：**
在 system prompt 开头增加：

```
背景：
- 用户是服装电商商家（平台：天猫/抖音/小红书）
- 诊断类型（result.type）共4种：traffic（流量型）、customer（客户型）、work（效率型）、unclear（不明确型）
- 生成的用户画像将用于：工作流推荐优先级、CTA 文案个性化、客服跟进话术生成
```

同时将规则 3 改为更具体的格式约束：

```
3. 格式：「[诊断类型标签] + [核心痛点（1个）] + [期望（1个）」，总字数控制在20-35字
   - 诊断类型标签必须从以下四个选一：流量型商家 / 客户型商家 / 效率型商家 / 探索型商家
   - 平台信息（天猫/抖音/小红书）若从答案中可推断则加入，否则省略
```

**理由：**
明确"服装电商商家"身份让 LLM 生成更有针对性的描述；明确四个诊断类型标签保证画像与系统路由语义一致；"20-35字"约束让 `max_tokens: 150` 有明确参照。

---

### 优化二：增加"不明确"类型的示例，并增加正负例对比

**改动方案：**
在示例中新增 2 条：

```
不明确型示例：
"刚入局服装电商、需求不清晰的探索期商家，期望明确最该先解决的问题"
"多平台运营、什么都想做但缺重点的迷茫型商家，期望找到核心突破口"

明确类型示例（补充）：
"小红书内容种草效果差、缺乏持续内容产出能力的内容型商家，期望AI辅助内容生产"
```

同时将规则 1 强化：

```
1. 若用户作答题目不足3道，画像应以探索型商家描述，重点输出"建议优先做XX"
2. 若用户作答≥4道，按诊断类型（traffic/customer/work）输出对应标签
```

**理由：**
补充 `unclear` 类型的示例让 LLM 有参照，减少模糊输出；加入"作答数量"判断逻辑让画像质量对部分作答用户也有保障。

---

### 优化三：增加结构化约束，确保画像包含 Q2（紧急度）和 Q4（AI 期望）信息

**改动方案：**
在规则中增加：

```
7. 画像必须整合以下两个维度的信息（若用户已作答）：
   - Q2（时间投入）：体现紧急程度（"每天处理"→高紧迫，"不到1小时"→低紧迫）
   - Q4（AI 期望）：直接引用用户的AI诉求（如"希望AI生成内容"、"希望AI自动回复客户"）
8. 输出格式示例（带结构化标记，供程序解析）：
   [流量型商家] [内容生产效率低+高紧迫] [期望AI生成高质量产品图]
   实际输出时不带括号，只输出句子内容
```

并在 user prompt 最后补充：

```
诊断答案包含Q1（痛点）、Q2（时间投入）、Q3（平台）、Q4（AI期望）、Q5（内容现状）。
请综合以上信息，生成一句包含"痛点+紧急度+AI期望"的用户画像。
```

**理由：**
当前画像缺失 Q2 的紧急度信息——同样说"内容生产效率低"，每天花1小时和每天花8小时的商家紧迫度完全不同，CTA 话术也应该完全不同。整合 Q4 的 AI 期望可以让画像直接驱动"工作流推荐"逻辑（如用户说"希望AI生成内容"→推荐 `content` action 工作流）。

---

## 4. 预期效果

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| **CTA 个性化** | 画像通用，前端需自行猜测用户类型 | 画像自带类型标签（流量型/客户型/效率型/探索型），可直接映射 CTA 话术 |
| **工作流推荐** | 画像与 workflowKey 无关联 | 画像整合 Q4 AI 期望，可直接推断推荐 `content`/`traffic`/`automation` action |
| **unclear 用户** | 生成模糊空话，无法引导 | 生成"探索型商家"描述 + 优先建议，帮助前端做二次分流 |
| **平台信息** | 偶有缺失，不稳定 | Q3 平台信息强制加入（如可推断），画像更具体 |
| **紧急度体现** | 完全缺失 | Q2 紧急度融入痛点描述（如"快崩溃了"→高紧迫 CTA） |
| **输出稳定性** | 结构随题目组合变化 | 固定格式：[类型] + [痛点+紧迫度] + [期望]，20-35字 |
